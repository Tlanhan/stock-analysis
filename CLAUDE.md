## 阶段定义

### 阶段 1: 本地讨论复盘(默认模式)
- **位置**: 当前文件夹 `G:/github/stock分析/`
- **内容**: 交易复盘、问题分析、规则设计、行情查询、情绪疏导
- **AI 行为**: 自由对话、深度分析、给建议、查数据、抓行情
- **❌ 禁止**: 在用户未说"更新 skill"之前,**绝不修改任何 `~/.claude/skills/` 下的文件**

### 阶段 2: 更新系统 skill
- **触发关键词**(用户说出任一即进入此阶段):
  - "更新 skill" / "写入 skill" / "加到 skill" / "记录进去" / "记录" / "记下来"
  - "升级 skill" / "改 skill"
- **位置**: `C:/Users/linguowei/.claude/skills/`
- **AI 行为**:
  1. 用 Read 确认目标文件结构
  2. 用 Edit/Write 改 SKILL.md
  3. 改完必须 **grep 验证**关键锚点 + 文件结构未破坏
  4. 总结改了哪些章节、位于哪些行号
- **❌ 禁止**: 改完直接 push GitHub(那是阶段 3 的工作)

### 阶段 3: 同步到 GitHub 仓库
- **触发关键词**(用户说出任一即进入此阶段):
  - "推送" / "push" / "同步到仓库" / "上传 GitHub" / "提交到仓库" / "上 git"
- **位置**: `G:/github/stock-skill/` → `origin/main`
- **AI 行为**:
  1. `cp` 系统 skill 文件覆盖到仓库对应目录
  2. `git diff` 确认变更范围
  3. **拆分 atomic commit**(每个独立逻辑一个 commit,不允许 mega commit)
  4. commit message 必须包含"为什么"(背景+治什么病),不允许 `update` / `fix` 这种空话
  5. `git push origin main`
  6. 报告 commit hash 和远程链接
- **❌ 禁止**: 在用户未说"推送"之前自动 push

## 🎯 交易决策强制调用 Skill 矩阵（最高优先级，2026-06-17 新增）

> **背景**：2026-06-17 用户多次表达交易意图（"后悔没做""我想买ETF""我要做159558"），但 Claude 只做了分析对话，**没有调用 `a-share-trigger-check`**——导致用户连续两天看好半导体设备却不敢建仓。根因：trigger-check 缺少"主动强制触发"规则，Claude 用"分析"替代了"执行框架"。
> **本规则**：用户表达任何交易意图时，Claude **必须先调用相关 skill 出执行框架，再做分析**。不允许用分析对话替代执行框架。

### 触发条件（命中任一即强制启动）

| 用户说的话（关键词）| 强制调用的 Skill + 模式 | 输出 |
|---------------------|----------------------|------|
| "我想买" / "我要做" / "可以做吗" / 明确提到标的代码 | **`a-share-trigger-check` 模式A** 写条件交易卡 | IF-THEN 条件卡（价格/仓位/止损）|
| "应该买吗" / "我有感觉" / "现在能进吗" | **`a-share-trigger-check` 模式B** 扳机检查 | 30秒 GO/NO-GO 决策 |
| "后悔没做" / "想做但没做" / "错过了" / "踏空" | **`a-share-trigger-check` 模式C** 失误复盘 | 结构化归因 + 明日改进 |
| "缩量" / "量价" / "昨天缩量" / "低开" | **`a-share-trigger-check` 模式I** 量价触发 | 四重确认 + 明日量价观察清单 |
| "复盘" / "今天怎么样" / "收盘分析" | **`a-share-daily-review`** | 完整10章复盘 + 量价清单 + 温度计 |
| "明天怎么做" / "明日策略" / "写卡" | **组合**：`a-share-trigger-check` 模式A + 情绪日历 + 海外催化 | 条件交易卡（含扰动检查）|

### 综合分析 Skill 矩阵（"调用一切可以调用的 skill"）

当用户问"是否应该交易"时，Claude **不只是调用 trigger-check**，而是按以下矩阵**组合调用多个 skill**，提供综合决策支持：

```
用户问"是否应该交易/买XX"
    ↓
[强制第1步] 调用 a-share-trigger-check（执行框架优先）
    ├─ 模式A 写条件交易卡（有明确标的时）
    ├─ 模式B 扳机检查（盘中临时起意时）
    ├─ 模式C 失误复盘（错过/踏空时）
    └─ 模式I 量价触发（缩量+低开时）
    ↓
[强制第2步] 加载背景文件（强制检查）
    ├─ Read 情绪日历 sentiment-calendar-YYYY-MM.md（检查扰动等级）
    └─ Read 海外催化 oversea-catalyst-YYYY-WNN.md（检查催化/风险）
    ↓
[按需第3步] 调用分析类 skill（用数据支撑决策）
    ├─ a-share-technical-analysis（技术面：均线/支撑压力/量价）
    ├─ a-stock-analysis（实时行情：qt_quote.py 拉一手数据）
    ├─ a-share-portfolio（持仓与仓位：当前仓位/资金/可灵活使用额度）
    └─ a-share-etf-selector（如涉及 ETF 选品）
    ↓
[输出] 综合决策报告 = 执行框架 + 背景检查 + 数据支撑
```

### 核心铁律

1. **执行框架优先于分析**：用户说"想交易"时，先出条件交易卡/扳机检查，再补分析。不允许先分析半天最后才问"要不要写卡"
2. **不允许用分析替代执行**：给出再多的行情分析、ETF对比、概率推演，如果最后没有一张可执行的条件交易卡，视为未完成任务
3. **组合调用而非单一调用**：交易决策需要多维支撑，至少覆盖"执行框架（trigger-check）+ 扰动检查（日历/催化）+ 数据（行情/技术面）"三个维度
4. **每个 skill 的输出必须嵌入最终回复**：不是"我调用了一下"就完了，要把 skill 的输出（条件卡/扳机判定/量价清单）实际写进给用户的回复里
5. **错过/踏空 = 强制模式C**：用户说"后悔""错过""踏空"时，必须用模式C的结构化归因表，不允许只做情绪安慰

### 🚫 反铁律

- ❌ 用户说"我想买XX"，Claude 只给分析不出条件交易卡
- ❌ 用户说"后悔没做"，Claude 只做情绪疏导不做模式C归因
- ❌ 用户说"应该买吗"，Claude 长篇分析后说"可以试试"（应该是明确的 GO/NO-GO）
- ❌ 调用了 skill 但没把输出写进回复（"我用了trigger-check"但回复里没有条件卡）
- ❌ 先分析后出卡（顺序反了，执行框架必须先行）

### 与其他强制规则的关系

本规则是**所有交易决策的前置层**，优先级高于：
- 情绪日历强制加载（交易决策时先出卡，再查日历补充扰动检查）
- 海外催化强制扫描（同理）

但本规则**不替代**数据时效性铁律——调用 skill 前仍必须先确认北京时间。

用户："我有点后悔早上没做159558，我觉得明天可能有回踩"
    ↓

## 📅 自动加载情绪扰动日历(强制规则)

### 触发条件
在本目录(`G:/github/stock分析/`)发生以下任一场景时,Claude **必须**先 Read 当月情绪日历文件:

- 用户问"本周/本月有什么事件"
- 用户准备做交易决策(任何买卖讨论)
- 触发 `a-share-trigger-check` 模式A/B/H 时

### 文件命名规则
**绝对路径**: `G:/github/stock分析/sentiment-calendar-{YYYY-MM}.md`

例如:
- 2026年6月 → `sentiment-calendar-2026-06.md`
- 2026年7月 → `sentiment-calendar-2026-07.md`

### 失效检查(机械执行)
每次进入本目录会话,Claude 自检:

| 情况 | 动作 |
|------|------|
| 当月文件不存在 | 提示用户:"本月日历缺失,需触发 a-share-sentiment-calendar skill 生成" |
| 当月文件存在但 `生成时间` 距今 > 25 天 | 提示用户:"日历可能过期,建议刷新" |
| 当月剩余 ≤ 5 天且无下月文件 | 提示用户:"下月日历未生成,建议提前准备" |
| 强扰动日(🔴)距今 ≤ 3 天 | **主动**在对话中提醒,即使用户没问 |

### 强制使用规则
1. **写卡铁律**: `a-share-trigger-check` 模式A 写明日卡前,**必须**读取当月日历,检查"明日"是否在 🔴 强扰动日中。若在,自动加入"不做清单"+ 降仓建议
2. **复盘铁律**: 收盘复盘时,若当日是 🔴 强扰动日,报告开头必须注明"今日波动可能源于 {events}"
3. **新发现写入**: 若用户提到日历中没有的新事件(政策窗口、突发会议),需建议用户更新日历文件

## ⏰ 数据时效性铁律（2026-06-15 新增，2026-06-18 增补会话启动校准）

### 背景
2026-06-15 用户发现 AI 在提供市场数据时多次混淆北京时间与美东时间，导致对美股开盘时间的判断错误。本规则强制 AI 在每次获取市场数据时**先确认当前北京时间**，再基于北京时间判断各市场的交易状态。

**2026-06-18 增补**：发现"每次获取数据前确认北京时间"仍是**被动规则**（用到时才查）。已发生事故：跨会话 AI 丢失"今天"的时间上下文，把 6/18 真实收盘数据错标成 6/17。根因是会话启动时没有**主动校准"今天"**。新增"会话启动校准"作为本铁律的前置层。

### 核心规则升级：会话启动必须校准"今天"（最高优先级前置层）

**每次会话开始的第一件事**，不是回应用户问题，而是先用 Python 确认并**明确告知用户**"今天是几月几号、星期几、当前北京时间几点、A 股处于什么交易时段（盘前/盘中/盘后/休市）"。这是所有后续讨论的时间基准。

```python
from datetime import datetime, timezone, timedelta
bj = timezone(timedelta(hours=8))
now = datetime.now(bj)
print(now.strftime('%Y-%m-%d %A %H:%M:%S'))
# 必须输出且告知用户：例如 "今天 2026-06-18 周四 16:17，A股已收盘"
```

**校准输出模板**（每次会话首条回复必须包含）：
> ✅ 已校准：**今天 2026-MM-DD 周X HH:MM**（北京时间），A 股状态：[盘前/盘中-上午/午间休市/盘中-下午/已收盘/休市]。

**为什么必须主动校准**：
- AI 跨会话丢失时间上下文，把"今天"当作训练数据里的某天 → 用旧数据分析"今天"
- 用户说"今天"/"昨天"/"明天"是相对时间，**必须先转绝对日期+星期**再处理
- 距上次会话 >1 天，主动提醒"情绪日历/海外催化文件可能需要刷新"
